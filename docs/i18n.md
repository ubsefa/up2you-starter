# Internationalization (i18n)

UP2YOU apps use locale JSON files and a pipe syntax in templates to translate labels, states, transitions, and validation messages.

## Locale Files

Each app can include locale files under `locales/`:

```text
my-app/
  locales/
    en.json
    tr.json
    es.json
```

The language code must be ISO 639-1 (e.g., `en`, `tr`, `es`) or with region (e.g., `en-US`, `tr-TR`).

### File Shape

Locale files are nested namespace objects with string leaf values:

```json
{
  "fields": {
    "title": "Title",
    "priority": "Priority",
    "assignee_user_id": "Assignee",
    "due_date": "Due Date"
  },
  "states": {
    "open": "Open",
    "in_progress": "In Progress",
    "done": "Done",
    "archived": "Archived"
  },
  "transitions": {
    "start": "Start",
    "complete": "Complete",
    "archive": "Archive"
  },
  "validation": {
    "title_required": "Title is required"
  },
  "charts": {
    "tasks_by_state": "Tasks by State"
  }
}
```

### Merging

When `GET /api/v1/_locales/{lang}` is called, Core merges locale files from **all apps** in the tenant into a single response. Later apps can override keys from earlier apps.

```bash
curl http://localhost:8080/api/v1/_locales/en \
  -H "X-Tenant-ID: 00000000-0000-0000-0000-000000000001"
```

Response:

```json
{
  "fields": {
    "title": "Title",
    "priority": "Priority"
  },
  "states": {
    "open": "Open",
    "in_progress": "In Progress",
    "done": "Done"
  }
}
```

## Using Locale Keys in YAML

YAML files reference locale keys using the `fields.`, `states.`, `transitions.` prefixes:

```yaml
columns:
  - field: title
    label: fields.title
  - field: current_state
    label_template: "{{current_state:states}}"
```

The renderer resolves `fields.title` to `en.json` → `fields.title` → `"Title"`.

---

## label_template

`label_template` formats values in table columns, select options, and detail views. It uses double-brace syntax with optional namespace pipe.

### Basic Field

```yaml
label_template: "{{title}}"
```

Inserts the raw `title` field value.

### Nested Field

```yaml
label_template: "{{patient_id.name}}"
```

Reads the nested `name` field from the `patient_id` reference object.

### Namespace Pipe

```yaml
label_template: "{{current_state:states}}"
```

The `:states` suffix translates the value through the `states` namespace in locale files:

- `current_state` value: `open`
- Lookup: `en.json` → `states.open` → `"Open"`

### Multiple Namespaces

```yaml
label_template: "{{patient_id.name}} - {{program_key:fields}} - {{current_state:states}}"
```

Produces: `"John Doe - Care Program - Active"`

### Available Namespaces

Common namespaces used in locale files:

| Namespace | Locale Key | Example |
| --- | --- | --- |
| `states` | `states.{value}` | `{{current_state:states}}` → `"Open"` |
| `fields` | `fields.{value}` | `{{program_key:fields}}` → `"Care Program"` |
| `transitions` | `transitions.{value}` | `{{transition:transitions}}` → `"Complete"` |

## Form Type Labels

Form fields use locale keys for labels:

```yaml
form:
  name: MyTodoCreate
  entity: Task
  fields:
    - name: title
      type: TextInput
      label: fields.title
      props:
        required: true
    - name: priority
      type: Select
      label: fields.priority
      options:
        - value: low
          label: Priority Low
        - value: medium
          label: Priority Medium
        - value: high
          label: Priority High
```

Option labels can also use locale keys:

```yaml
        options:
          - value: low
            label: priority.low
          - value: medium
            label: priority.medium
          - value: high
            label: priority.high
```

With `en.json`:

```json
{
  "priority": {
    "low": "Low",
    "medium": "Medium",
    "high": "High"
  }
}
```

## Workflow and Transition Labels

Workflow transitions use the `transitions` namespace:

```yaml
transitions:
  complete:
    from: [in_progress]
    to: done
    permissions: [user, admin]
```

The renderer looks up `transitions.complete` in locale files for the button label.

In `en.json`:

```json
{
  "transitions": {
    "complete": "Mark Complete"
  }
}
```

---

## View and Component Titles

View names and chart titles use locale keys:

```yaml
view:
  name: MyTodoTasks
  layout: Stack
  components:
    - type: Chart
      props:
        title: charts.tasks_by_state
```

In `en.json`:

```json
{
  "charts": {
    "tasks_by_state": "Tasks by State"
  }
}
```

---

## EntitySelect and AppMemberSelect Labels

Dynamic selects use `label_template` for option display:

```yaml
- name: patient_id
  type: EntitySelect
  props:
    options_query: all_patients
    label_template: "{{name}}"
    required: true
```

```yaml
- name: assignee_user_id
  type: AppMemberSelect
  required: true
```

`AppMemberSelect` uses the member's display name from the Platform member list.

---

## Validation Messages

User-facing validation messages can reference locale keys. For form fields, use `props.required` to indicate required fields:

```yaml
form:
  name: TaskCreate
  entity: Task
  fields:
    - name: title
      type: TextInput
      label: fields.title
      props:
        required: true
```

For workflow transitions, guard expressions can include a `guard_message` that references a locale key for user-facing error text.

In `en.json`:

```json
{
  "validation": {
    "title_required": "Title is required"
  }
}
```

## Adding a New Language

To support a new language:

1. Create `locales/{lang}.json` in your app package:

```json
{
  "fields": {
    "title": "Başlık",
    "priority": "Öncelik"
  },
  "states": {
    "open": "Açık",
    "in_progress": "Devam Ediyor",
    "done": "Tamamlandı"
  },
  "transitions": {
    "start": "Başlat",
    "complete": "Tamamla"
  }
}
```

2. Request the locale from the API:

```bash
curl http://localhost:8080/api/v1/_locales/tr \
  -H "X-Tenant-ID: 00000000-0000-0000-0000-000000000001"
```

3. Your frontend renderer should switch the locale based on user preference and use the returned strings for all labels.

---

## Renderer Responsibility

Core and SDUI Gateway **do not** translate labels in API responses. They return raw data and locale files separately. The **frontend renderer** is responsible for:

1. Fetching locale strings via `GET /api/v1/_locales/{lang}`.
2. Resolving `fields.{key}`, `states.{key}`, `transitions.{key}` references in view schemas.
3. Applying `label_template` pipes to translate values.
4. Switching locales when the user changes language preference.

The hosted Platform renderer handles this automatically. For custom frontends, implement a simple i18n layer that maps locale keys to translated strings.
